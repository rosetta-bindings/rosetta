# Changelog

All notable changes to **rosetta** are documented here.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/). This project has not yet cut a tagged release, so entries are grouped by date rather than by version number. Dates are `YYYY-MM-DD`.

## [Unreleased]

### Changed
- **Plain `rosetta_gen` emits incrementally and guides the next step** — `rosetta_gen manifest.json` now writes `bindings.h` / `<target>.cpp` / `CMakeLists.txt` only when their content actually changed (`write_file_if_changed`), so an already-configured `generated/build` recompiles nothing after a no-op re-emit, and prints `unchanged:` instead of `wrote`. Two footguns of the manual three-step workflow (`rosetta_gen` → `cmake --build` → `./generator`) get a printed note: when `generated/build` has no `CMakeCache.txt` (fresh checkout, or a `--clean` removed the whole project dir) the full `cmake -S … -B …` configure command is spelled out — `cmake --build` alone fails there; and when the emitted sources DID change while a configured build dir exists, a warning reminds that the generator binary is now stale and, run without rebuilding, would silently re-emit the previous bindings (exactly the failure mode observed after renaming a bound function: the old binary kept emitting the old API while every backend build failed against the new headers). `--build` shares the incremental emission (faster no-op re-runs) and needs no notes — it configures and rebuilds itself.

### Added
- **Manifest `build_type` / `optimization` — build configuration for every generated CMakeLists** — two new optional top-level manifest fields bake a build configuration into every compiled backend's generated `CMakeLists.txt` (thin and `-expanded` alike; the text-only backends ignore them). `"build_type"` (`Debug` | `Release` | `RelWithDebInfo` | `MinSizeRel`, case-insensitive) is emitted as a *default* inside `if(NOT CMAKE_BUILD_TYPE)`, so `-DCMAKE_BUILD_TYPE=...` at configure time still wins. `"optimization"` (`-O0`…`-O3`, `-Os`, `-Oz`, `-Og`, `-Ofast`; leading `-` optional) is emitted as `add_compile_options` / `add_link_options`, which land *after* the build type's per-config flags on the command line — so it overrides the `-O` level the build type implies (and reaches the wasm link, where emscripten optimizes at link time too). Plumbed as `GenerateOptions::build_type` / `::optimization` → `GenContext` → a new `{{BUILD_CONFIG}}` placeholder that `render_meta()` splices in right after each template's `set(CMAKE_CXX_STANDARD_REQUIRED ON)` line; both fields empty ⇒ byte-identical output to before.

### Fixed
- **Expanded backends: nested class types are spelled fully qualified** — `display_string_of` prints a class nested inside another class by its bare identifier (`Planes` for `stressinv::NodalPlane::Planes`), and the emitted code opens namespaces (`using namespace`) but cannot open classes, so any constructor or trampoline signature naming a nested type (`NodalPlane(vector<Serie>, Planes, Planes, double)`, `StressDomain(…, const Axis&, …)`) failed to compile. `exact_spelling()` now re-qualifies the bare token with the full parent chain — enclosing namespaces *and* classes (new `qualified_class_name<T>()`, which unlike `class_namespace()` keeps walking through outer class scopes). Subtlety worth recording: the type reaches the helper through `std::remove_cvref_t`, an alias template, and `^^U` then reflects the *alias* — whose `parent_of` is the alias's own scope, not the class's owner — so the reflection must be `dealias`ed before walking parents.
- **Constructor parameters of std-namespace types are spelled by value — dangling-view fix** — the ctor spellings in `ctor_param_cpp` were exact (`const std::vector<double>&`), and every expanded backend reproduced them verbatim in `init<...>`. Two problems: bindings pass *caster temporaries* for std types (a Python list / JS array converts to a temporary `std::vector`), so a constructor that *stores* the reference captured freed memory — `Serie(values, 3).count()` returned garbage because nanobind's first-match dispatch handed every call to the `const std::vector<double>&` **view** ctor instead of the owning `&&` one; and the `const&`/`&&` pair bound as two indistinguishable overloads. New `ctor_param_spelling()`: a parameter whose decayed type lives in namespace `std` is spelled by value (routes construction to the move overload — never dangles, no extra copy), while user class types keep their exact spelling (they arrive as references to live registered objects, and a class may deliberately store that reference, e.g. a domain object referencing its model). Constructors whose spellings collapse to the same list dedupe to one IR entry.
- **The walk skips `std::initializer_list` constructors** — no target language can produce one (no marshaller in any backend), its exact spelling doesn't round-trip (`initializer_list<double>`, unqualified), and such a constructor always shadows an equivalent `std::vector` overload. `is_exportable_constructor()` now rejects any ctor with an `initializer_list` parameter, in all backends at once (`initializer_list` also added to `qualify_std`'s table for the residual method-signature case).
- **Node: a non-default-constructible class is constructible from script** — the `Wrap` constructor's parameterized path default-constructed the inner object then *assigned* the ctor-table result into it, so a class whose only constructors take arguments (every stressinv data class: `Joint`, `Striation`, `GPS`, `StressDomain`…) threw "this class cannot be constructed directly" — previously only reachable as a member-object alias. The Wrap now falls back to building the object *straight from the ctor-table entry into fresh storage* (move/copy construction, no default-construct-and-assign), gated on the new `GenClass::copy_or_move_constructible`; the emitter registers entries for a default-constructible class when it is assignable (unchanged) and for a non-default-constructible one when it is move- or copy-constructible.

### Changed
- **Expanded backends emit `CMAKE_CXX_STANDARD 20`** — the nanobind / python / lua / qt / qml expanded CMake templates still pinned C++17 (node / wasm were already 20), so a bound library using C++20 in its headers (`std::span`, concepts) failed to compile. All expanded templates now emit 20.

All four fixes verified end-to-end on the stressinv manifest (20 classes, 14 free functions): nanobind, node and wasm backends build with stock toolchains and run the same Andersonian inversion example; rosetta test suite 56/56.

### Added
- **`rosetta_gen --build` — the whole manifest pipeline in one command** — `rosetta_gen --build manifest.json` emits the generator project, builds and runs it (clang-p2996), then compiles every backend the manifest declares with the right per-backend build shape: plain CMake for most, `npm install && npm run build` for node, `emcmake` for wasm, and the `dotnet build` / `mvn package` second stages for csharp / java. A backend whose toolchain is absent (no emsdk, no julia, no dotnet…) is skipped with a note rather than failing the run; a failed build is reported in the end summary and `--build` exits non-zero after trying the rest. Options: `--only`/`--skip` backend lists, `--clang-p2996-root` (forwarded to every CMake configure and to cmake-js), `--qt-dir`, `--cmake-arg` passthrough, `--gen-dir`/`--bindings-dir`, `--jobs`, `--fresh`. Implemented inside the tool itself (`std::system` + `std::filesystem`, no shell script) so the same binary drives the pipeline on Linux, macOS and Windows — including multi-config (MSVC) generator-binary lookup. Verified end-to-end on `examples/geom-expanded` (expanded backends incl. the csharp `dotnet build` stage) and `examples/geom-lib` (thin python + json). The companion **`rosetta_gen --clean manifest.json`** removes everything the pipeline generated — the generator project dir (`gen/` or `generated/`), the generator binary, and the manifest's backend folders under `bindings/` — never the manifest, and never a folder missing the "Generated by rosetta" stamp (a hand-written folder sharing the name survives); files a user parked under `bindings/` are kept, with a note.
- **`rosetta_gen` builds on Windows** — the `user_sources` glob expansion no longer uses POSIX `<glob.h>`; a `std::filesystem`-based expansion with the same semantics (`*`, `?`, `[...]` per path component, wildcards don't match a leading dot, unmatched pattern → empty + warn) replaces it, removing the tool's only non-portable dependency.
- **Per-class `"final": true` manifest flag — suppress the trampoline** — a class marked final is treated as non-overridable from the host language: no `Py_T` / `Js_T` trampoline is generated even when it has public virtual methods (they still bind as ordinary callable methods). Two motivations: skipping useless shims for classes nobody script-subclasses, and — the geogram case — making a virtual-bearing class eligible as a **node member-object property**: the aliased wrap stores a `T*`, which requires the wrapped type to be `T` itself, so `GEO::MeshVertices` / `MeshFacets` (public `delete_elements` / `permute_elements` / `pop` virtuals from the `MeshElements` interface) can now be reached as `mesh.vertices` / `mesh.facets` under node too. Flows as `ClassEntry::final_` → generated driver `opt.final_classes` → `GenClass::is_final`; `virtual_methods()` / `node_virtual_methods()` return empty for a final class, which every trampoline-emitting backend already keys on. (`tests/member_object.cpp`: with/without-final node render pair + python.)
- **Foreign sequence containers (`rosetta::is_sequence` trait + manifest `sequences` field)** — a library's own vector type (`GEO::vector<T>`, an aligned/pooled vector) now crosses the boundary like a `std::vector` of its element. Registration is opt-in: `"sequences": ["GEO::vector"]` in the manifest (rosetta_gen emits the `rosetta::is_sequence<GEO::vector<T>>` specialization into `bindings.h`), or the specialization written by hand for programmatic use (new header `rosetta/sequence.h`; requirements: default-constructible, `value_type`, `size()`, `resize(n)`, `begin()`/`end()`; elements arithmetic / `bool` / `std::string` / bound enum). The IR keeps `kind` "unknown" (the `is_pointer` / `is_callback` pattern — backends that don't opt in keep skipping the type) and records `GenType::is_sequence`, the element, and a **composed qualified spelling** `seq_cpp` (`GEO::vector<double>` — `display_string_of` prints template names unqualified, so the spelling is rebuilt from the enclosing namespaces + the template identifier + the element). The opted-in backends — **python- / nanobind- / node- / wasm- / lua-expanded**, plus **typescript** declarations (`number[]`) — marshal by COPY through a `std::vector<element>` boundary inside an emitted adapter (python/nanobind/wasm/lua: lambda; node: a namespace-scope free function the runtime's `fn_traits` introspect, bound through `ext_method` / `call_static` / `napi_free_entry`; wasm registers the boundary vector; lua emits a `sol::overload` pair accepting BOTH container userdata — what a bound sequence return pushes — and plain tables via `sol::nested`). Three deliberate consequences: a **mutable `Seq&` parameter binds input-only** (the adapter's local is a real lvalue — geogram's `assign_points(vector<double>&, dim, steal)` works, `steal` steals from the adapter's copy); an **overload set whose surviving (first-declared) IR entry is the sequence overload binds** — the adapter calls by NAME with concrete arguments instead of spelling the ambiguous `&T::name`, which is exactly the `MeshVertices::assign_points` / `MeshFacets::assign_triangle_mesh` shape; and a **virtual method naming the container keeps its trampoline override off** (`sig_type_bindable` now rejects registered sequences — the exact spelling can't round-trip through `qualify_std`, which would mis-qualify it to `std::vector` and break the override; the method still binds callable — this was the "trampoline spelling of GEO::vector virtuals" blocker that kept `MeshVertices` / `MeshFacets` out of geogram's manifest). Sequence-typed public fields bind as copying properties (python/nanobind/wasm/lua). Verified end-to-end on geogram: `mesh.facets.assign_triangle_mesh(3, coords, tris, false)` from a Python list builds the mesh, round-tripping exactly against the flat-array extensions (`tests/sequence.cpp`, 8 gtests; suite 54/54).
- **Member-object properties on wasm-expanded** — the non-copyable member object of a bound class (`GEO::Mesh::facets`) now binds under embind too, as a **borrowed-handle getter method** (`mesh.facets().nb()`): embind properties copy, so the emitter binds `.function("facets", +[](Mesh &s) { return &s.facets; }, emscripten::allow_raw_pointers())` — a non-owning pointer handle to the member living inside the parent. Unlike pybind's `reference_internal` nothing pins the parent (embind has no keep-alive policy) — the handle dangles if the mesh is destroyed first; same clash rule as python (a method of the same name wins). **lua-expanded needed no change** — sol2 pushes a class member by reference, so the member object already rode the ordinary readonly member-pointer path — but is now covered by tests. With this, all four runtime backends expose `mesh.facet_corners`-style sub-objects, except node's trampolined-class case (`MeshVertices` / `MeshFacets` have virtuals; node's aliased wrap stores a `T*`, not a `Js_T*` — unchanged).
- **Member-object properties — a class's non-copyable member objects become reference properties** — a public data member whose type is another *bound* class but is not copyable (it holds a back-reference to its owner, like `GEO::Mesh::vertices` / `facet_corners`) used to be skipped wholesale by the copyability gates; the geometry it carries was only reachable through hand-written extension functions. It is now bound as a **read-only property returning a reference to the real member**, with the parent object kept alive for as long as any child handle exists. **python-expanded** emits `def_property_readonly` with a reference-returning lambda and `py::return_value_policy::reference_internal`. **node-expanded** reworks the runtime `Wrap` to hold its object **by pointer in one of two modes** — *owned* (the ordinary `new Cls()` path) or *aliased* (the new `get_member_object<&Cls::member>` accessor wraps the member's address and pins the owning JS object in a persistent reference); a pleasant side effect is that a **non-default-constructible class is now wrappable at all** (its JS constructor throws "reached as a member of another object" instead of failing to compile), where the old by-value `Tramp inner` storage required default-constructibility of every bound class. **typescript** declares the property (`readonly facet_corners: MeshFacetCornersStore;`). On a name clash (e.g. a manifest extension already named like the field) the method wins and the property is skipped; node additionally restricts the alias to untrampolined member classes (the stored pointer is a `T*`, not a `Js_T*`). **wasm- / lua-expanded** still skip such fields (follow-up). Verified end-to-end on geogram: `mesh.facet_corners.nb()` / `.vertex(c)` alias the live store in Python and Node, including keep-alive after the last mesh reference is dropped and GC runs (`tests/member_object.cpp` plus geogram's `test_member_object.{py,js}`).

### Fixed
- **Overload sets no longer break the expanded builds** — the member walk dedups overloads by identifier (one IR entry survives), but the expanded emitters spell the bare `&T::name` member pointer, which is **ambiguous** for an overload set — binding any class with a const/non-const accessor pair (`GEO::MeshVertices::point_ptr`, `point_coordinates`, …) failed to compile. A new reflection-time `GenMethod::is_overloaded` flag (more than one exportable member function of the declaring class shares the name) lets **python- / node- / wasm- / lua-expanded** skip the whole set, and **typescript** hide it, keeping the declarations aligned with the runtime surface. Conservative by design — overload selection by explicit signature in the manifest is the planned lift.

### Changed
- `node_runtime.h` now holds only the documented declarations (plus the small compile-time trait types they need); the implementations — the conversion helpers, the override plumbing and all of `Wrap`'s member definitions — moved to `backends/inline/node_runtime.hxx`, included at its bottom (the `generate.h` / `inline/generate.hxx` layout convention, as previously done for `annotate.h`).

### Fixed
- **Out-of-line side-car: scientific notation in numbers** — the consteval JSON parser had no exponent support, so a `"range": [1e-10, 1e-6]` (a routine solver tolerance) parsed as `[1, 0]` **and silently derailed every key after it** — later members lost their doc/range/combobox entirely. `to_double` / `number()` now consume `1e-10`-style exponents.
- **Emitted range bounds lose tiny magnitudes** — `num_str` rendered bounds via `std::to_string`, which collapses `1e-10` to `0.000000`; generated validating setters and docs (all backends) then carried a wrong bound. Now formatted with `%g` (`1e-10` stays exact).

### Added
- **Four new widget hints: `color`, `multiline`, `radio`, `file`** — implemented across all three UI inspector backends (Dear ImGui, Qt Widgets thin + expanded, QML thin + expanded), inline (`rosetta::widget::color` …) and out of line (`"widget": "color"` …). `color`: a string field holding `"#rrggbb"` renders as a color picker (ImGui `ColorEdit3`, Qt swatch button + `QColorDialog`, QML swatch + `ColorDialog`) whose edits write back as hex. `multiline`: a string as a multi-line text area (`InputTextMultiline` / `QTextEdit` / `TextArea`). `radio`: renders the field's `combobox{}` choices as a radio-button group instead of a drop-down (pair the two annotations). `file`: a path string as text entry + "…" browse button opening the platform file dialog (`QFileDialog` / QML `FileDialog`; ImGui, which ships no dialog, shells the native one — `osascript` on macOS, `zenity` on Linux — and degrades to plain text entry elsewhere). `examples/imgui`'s `Algo` gained `precond` / `plotColor` / `meshFile` / `notes` fields exercising all four, and its manifest now also emits `qt-expanded` / `qml-expanded` so the three inspectors can be compared on the identical annotated surface. (Known environmental issue, not rosetta's: Qt 6.8.3 + recent Xcode SDKs fail to link with `ld: framework 'AGL' not found` — use Qt 6.5.3 or a Qt built for the newer SDK.)
- **Out-of-line side-cars reach annotation parity** — the `.ann.json` format now also carries `label` (display-name override), `button` (call-button caption for a method) and `widget` (`"slider"` / `"spin"` / `"checkbox"` / `"textfield"` UI hint), completing the set (`doc` / `range` / `readonly` / `combobox` were already supported). A stock-C++ header + side-car now drives *exactly* the same UI as inline `[[=rosetta::...]]` annotations. Reminder: the side-car is baked into `bindings.h` when `rosetta_gen` runs — re-run it after editing the JSON.
- **Dear ImGui backend (`imgui-expanded`)** — the immediate-mode counterpart of the `qt-expanded` inspector: a self-contained desktop app (GLFW + OpenGL 3; Dear ImGui and, when not installed, GLFW are fetched automatically at configure time — imgui ships no CMakeLists, so its sources compile into the target) with one tab per default-constructible bound class. Numeric fields are drag boxes, or *clamping sliders* when a `range` annotation bounds them (the slider enforces the range by construction; `widget::slider` / `widget::spin` hints honored); bools are checkboxes; strings are text inputs (`combobox` choices become a combo); enum fields get a combo over the reflected enumerators; numeric `std::vector` members get a collapsible per-element editor with +/− resize; `readonly` greys the widget out; `doc` becomes a "(?)" hover tooltip; scalar-marshalable methods (same rule as the Qt inspector) get a call button showing the stringified result (`button` annotation labels it, exceptions are caught and displayed) — bare for a zero-argument method, inside a tree node holding one input per parameter otherwise. Because ImGui is immediate-mode, the generated `draw_<Class>()` functions run every frame and bind widgets *directly to the live members* — the object is the single source of truth, no copies, no signal plumbing. The widget kit lives in the new header-only, reflection-free `rosetta/visitors/imgui_runtime.h` (the ImGui `qt_widgets_runtime.h`); the generated app builds with a stock C++20 compiler. `ROSETTA_IMGUI_FRAMES=N` auto-exits after N frames for headless smoke tests / CI. The generated CMake takes `-DROSETTA_IMGUI_CPP26=ON` to build with the clang-p2996 toolchain instead (`-fannotation-attributes`, fork libc++, `-fblocks` on macOS) — needed only when the bound headers carry *inline* `[[=rosetta::...]]` annotations, since the inspector `#include`s them; out-of-line side-cars keep the stock build. The `widget::spin` / `widget::textfield` hints select a typed input box (clamped when ranged) over the slider/drag default. `examples/geom-expanded` gained the target (`geom_imgui`, out-of-line annotations, stock build), and the new **`examples/imgui`** binds a stock-C++ copy of the shared `Algo` demo class with every annotation out of line in `Algo.ann.json` (`doc` / `range` / `combobox`), so the inspector builds with a stock C++20 compiler (`./run.sh`; the inline-annotated original stays at `examples/Algo.h`).
- **Julia backend, expanded variant (`julia-expanded`, CxxWrap / jlcxx)** — brings Julia up to the python-expanded feature level, and fixes the thin backend's biggest gap: building against the *stock* libc++ (no clang-p2996) lets `<jlcxx/stl.hpp>` compile, so **`std::vector` members, parameters and returns are fully bound** — including vectors of bound classes (`getSurfaces() → StdVector` of `Surface`), which the thin `julia` target must skip entirely (fork libc++ gap). Registration is two-phase (all `add_type`s, then members — jlcxx resolves signature types eagerly, so `Model::addSurface(Surface&)` would otherwise fail when `Surface` is registered later). Covers constructors with exact spellings (plus an `ArrayRef` factory variant so `Surface([0.0,0,0,…], Int32[0,1,2])` takes plain Julia `Vector`s, zero-copy view), fields as `name(obj)` / `name!(obj, v)` with `readonly` / validating `range`, methods with exact-signature `static_cast` disambiguation (dropping `noexcept`, which jlcxx's templates don't match), an `ArrayRef` overload per numeric-vector parameter, statics as module functions, extension methods, single inheritance via `jlcxx::SuperType` + `julia_base_type`, free functions, raw pointers to bound classes, and the copyability gates (an lvalue-ref class/vector return is wrapped in a by-value-copying lambda, matching pybind's `automatic` policy). Enum constants are prefixed with the enum name (`Kind_Point`) — Julia module constants share a flat namespace with the types, so a bare `Point` enumerator collides with a bound `Point` class ("Duplicate registration"). Emits `auto_jlcxx.cpp`, a CMakeLists that locates JlCxx through `CxxWrap.prefix_path()`, and a `<lib>.jl` loader module. Callbacks (`std::function` params) stay skipped — jlcxx has no Julia-closure → `std::function` conversion. `examples/geom-expanded` gained the `julia-expanded` target (`jlgeom`) and `example_julia.jl` exercising the vector round-trips, range validation and the free function.
- **Lua backend (`lua-expanded`, sol2)** — a 25th backend: fully-expanded [sol2](https://github.com/ThePhD/sol2) bindings emitting a `require`-able C module (`luaopen_<name>`, built as `<name>.so`) for Lua 5.1–5.4 / LuaJIT with a stock C++17 compiler; sol2 is fetched automatically at configure time (an installed package wins). Covers constructors (full runtime overload resolution — no embind-style one-per-arity limit), fields (readonly / validating `range` properties), methods with exact-signature `static_cast` disambiguation, statics, extension methods (free function bound as instance method), enums (`new_enum`, read-only), free functions, bound-class raw pointers (non-owning), `std::function` callbacks (a Lua function converts natively — no adapter shim), and multiple inheritance via `sol::bases<...>` (embind is single-base). `std::vector` crosses both ways: a returned container is reference-semantics userdata (`#v`, `v[i]`, in-place mutation of the C++ object), and any signature naming a vector gets a second, table-accepting overload (`sol::nested`) so plain Lua tables work too — writable vector *fields* get a type-branching setter accepting either form. An lvalue-ref class return is pushed by reference (no copy), so non-copyable classes keep more surface than under pybind/embind. Note: sol2 does not support Lua 5.5 (changed C API); the generated CMake prefers a 5.4 installation and warns on 5.5+, and the emitted `luaopen_` guards with `luaL_checkversion` so loading the module into a mismatched interpreter (e.g. Homebrew's default `lua`, now 5.5) raises a clean "version mismatch" error instead of segfaulting. `examples/geom-expanded` gained a `lua-expanded` target (`luageom`) and an `example_lua.lua` exercising table-constructed Surfaces, the `std::function` callback, range validation and the enum table.
- **Extension methods (`extensions` manifest class field)** — a class entry may list free functions whose **first parameter is `Cls&` (or `const Cls&`)**; each is exposed as an ordinary *instance method* of the bound class, with the receiver dropped from the exposed signature and the method named after the function's unqualified identifier. This is the escape hatch for a library whose own members can't cross the boundary — raw-pointer accessors (`GEO::Mesh`'s `point_ptr()` returns a bare `double*`), attribute-template lookups, overloaded helpers (`mesh_load`/`mesh_save`) — without a hand-written wrapper class: the glue shrinks to stateless free functions and scripts keep holding the real C++ objects (`mesh.set_surface(coords, tris)` calls `georo::set_surface(mesh, …)`). `generate()` validates the receiver parameter and splices the function into the class IR (`GenMethod::is_extension` / `ext_qualified` / `ext_header`; `GenerateOptions::extensions` / `GenExtension` for programmatic use). **python-expanded / nanobind-expanded** bind the free function directly (pybind/nanobind treat a `T&`-first free function as a method); **wasm-expanded** likewise via embind; **node-expanded** calls it through a new `Wrap::ext_method<FP>` that passes the wrapped object as receiver; the text backends (**typescript / markdown / html**) render it as a normal method. Backends that can only emit member pointers (qt / qml / C# / Java / REST) skip extensions explicitly; the thin (reflection-re-running) backends don't see them.
- **Copyability captured in the IR, and emitters gate on it** — every runtime backend copies at some boundary (pybind's `def_readwrite` setter and its `automatic` policy on an lvalue-ref return, embind's `.property` accessors and class returns, N-API's `to_napi` construct-and-assign), so a class whose copy is deleted made the *generated code fail to compile*. The reflection walk now records `std::is_copy_constructible` / `is_copy_assignable` per type (`GenType::copy_constructible` / `copy_assignable`, false for incomplete types), whether each parameter is an lvalue reference and whether it is a *mutable* one (`GenParam::is_ref` / `is_mutable_ref`), whether a method returns an lvalue reference (`GenMethod::ret_is_ref`), and whether the class itself is assignable (`GenClass::copy_or_move_assignable`). The **python- / nanobind- / node- / wasm-expanded** emitters then skip exactly what they could not compile — a non-copyable public field (e.g. `GEO::Mesh::vertices`, a member object holding a `Mesh&` back-reference), a method returning a reference to a non-copyable class (`Mesh::get_subelements_by_index`), a by-value parameter of one, and (node/wasm) a *mutable non-class reference* out-parameter (`std::string&`, `index_t&` — the converted argument is a temporary, which can't bind to a non-const reference). **typescript** applies the same visibility rules so the declarations match the runtime surface. A by-reference class parameter is exempt everywhere — it never copies. Net effect: a class like `GEO::Mesh` binds directly as an opaque handle plus its marshalable methods instead of breaking the build.

- **Per-target `link_options` manifest field** — an object-form target may list extra linker flags (`{"lang": "wasm-expanded", "link_options": ["-lnodefs.js"]}`), emitted as `target_link_options(... PRIVATE ...)` in *that* target's generated CMakeLists only. Per-target — unlike `compile_definitions` — because link flags are toolchain-specific: e.g. emscripten's nodefs library is needed when the bound code mounts NODEFS under Node, but would break a native link. (`TargetSpec::link_options` / `GenContext::link_options` for programmatic use.)
- **`compile_definitions` manifest field** — preprocessor definitions (`"NAME"` or `"NAME=VALUE"`) applied to the generator driver *and* every compiled binding target via `target_compile_definitions(... PRIVATE ...)`, so the reflection walk, the bound headers and the `user_sources` all see the same preprocessor state. Typical use: a third-party library's configuration switches, e.g. geogram's `GEOGRAM_USE_BUILTIN_DEPS` (use the vendored libMeshb / rply / zlib instead of system libraries) and `GEOGRAM_WITH_HLBFGS` (compile the HLBFGS optimizer in). Ignored by the text-only backends. (`GenerateOptions::compile_definitions` / `GenContext::compile_definitions` for programmatic use.)
- **C sources in `user_sources`** — entries may now be `.c` files (e.g. vendored zlib / rply / libMeshb / OpenNL); the generated binding CMakeLists then calls `enable_language(C)` automatically so they compile alongside the C++ binding (native backends and the wasm templates alike).
- **Node / WebAssembly: JS-function callbacks are marshalled into `std::function` parameters** — a method or constructor taking a `std::function<R(A...)>` (e.g. `UserRemote(fn, isStress)` for a per-point far-field stress, `Solver::onMessage` / `onEnd` progress callbacks) now accepts a host-language function, wrapped into the C++ `std::function` and invoked synchronously (valid for callbacks fired during a host-initiated call such as a `solver.run()`). The neutral IR gained `GenType::is_callback` plus a decomposed `callback_sig` (`[0]` = return type, kind `void` when none; `[1..]` = parameter types), populated in `type_descriptor`. A callback is bound only when its whole signature is convertible — scalars, `bool`, strings, enums, `std::array` (element-wise), vectors of such, and bound classes — so a callback naming a raw pointer, an unbound by-value type, or a nested callback (e.g. `Triangle*`, `Eigen::VectorXd&`, an unbound `StepResult&`) is still skipped rather than breaking the build. **Node** makes `napi_supported<std::function<…>>` true for a convertible signature and materializes the callback with `from_napi`, keeping the JS function alive in a shared `Napi::FunctionReference`; `std::array` gained an N-API array conversion. **WebAssembly (expanded)** emits an `emscripten::val` adapter — a factory-lambda constructor `+[](emscripten::val, …) -> T*` or a receiver-first method lambda — plus a `rosetta_wx` helper (`to_val` / `from_val` / `make_fn`) inlined into the generated TU (embind can't marshal a `std::function` parameter directly). (Python already handled this via `pybind11/functional.h`.)
- **Node / WebAssembly: raw pointers to bound classes are marshalled** — a method returning `T*` (e.g. `Model::addSurface() → Surface*`) or taking one (e.g. `Model::addRemote(BaseRemote*)`, `Postprocess::burgersFor(Surface*)`) is now bound, as a *non-owning* handle to the C++ object (the C++ side keeps ownership). The neutral IR gained `GenType::is_pointer` (with the pointee identifier). **WebAssembly (expanded)** emits `emscripten::allow_raw_pointers()` on any signature naming a raw pointer, gated so only a pointer to a *bound* pointee is accepted (a vector-of-pointer is still skipped — embind has no `register_vector<T*>`). **Node** wraps a returned `T*` by retargeting a fresh wrapper's `self_` at the external object; `Wrap` now holds its owned value in a `unique_ptr` (via a heap-constructing ctor table) so a non-default-constructible type (`SeidelSolver(Model&)`) or an abstract referenced-only base (`BaseRemote`) works, and all member access goes through a `self_`-based `obj()` so owned and referenced wrappers behave identically. A pointer to a forward-declared class is left unsupported (completeness is required).
- **Python / WebAssembly: C++ inheritance is registered** — a bound class now declares its bound base classes to the target framework, so a derived instance is accepted wherever a base pointer/reference is expected (e.g. passing a `UserRemote` to `Model.addRemote(BaseRemote*)`, or a `Coulomb` where a `Tic` is wanted) and the hierarchy is visible to the host language (`issubclass` / prototype chain). Direct public bases are reflected into `GenClass::bases` (via `std::meta::bases_of`) and each backend filters them to the bases that are themselves bound. **Python** threads them through `bind_pybind` into `py::class_<T, Bases..., Trampoline>` (multiple bases; trampolines compose). **WebAssembly (expanded)** emits `emscripten::class_<T, emscripten::base<Base>>` (embind supports single inheritance, so the first bound base is used). Bases must be registered before their derived — which follows from manifest order; a base that isn't bound is simply omitted. (Node is not yet covered: node-addon-api `ObjectWrap` does not compose C++-base inheritance across separately-wrapped types without a dedicated prototype-chain shim.)
- **`rosetta_gen --init` flag** — writes a fully-commented example `manifest.json` (default `./manifest.json`, or a path argument) so a new project can start from a working template instead of the docs. The example exercises the common fields: `cpp26_*` toolchain overrides, an array-form `user_include`, `rosetta_include`, `generator_name` / `module_name`, `user_sources`, a representative spread of `targets`, and one example class and one example function. It **refuses to overwrite an existing file** — warning and exiting non-zero instead — so a hand-written manifest is never clobbered.
- **`user_sources` manifest field** — a list of user `.cpp` files compiled directly into every generated binding target via `target_sources(...)`. Use it when the bound headers only *declare* the API and you want the bodies built into the binding rather than linked from a pre-built `user_lib`. Works alongside or instead of `user_lib`; ignored by the text-only backends. Entries may be **shell globs** (e.g. `src/algorithms/*.cpp`), expanded at generation time; results are sorted and de-duplicated.
- **Multiple include directories** — `user_include` now accepts an array of directories (a class `header` is searched across all of them, like a compiler's `-I` order), in addition to the existing single-string form.

### Fixed
- **Generator link no longer requires the bound class's constructor** — `IRVisitor::field` default-constructed a `T tmp{}` to read *every* field's default value, odr-using `T`'s constructor; since the driver links no user library, a class with a header-declared constructor (`GEO::Mesh`) failed at link time. The instance is now built only for field kinds `default_value_str` can actually render (arithmetic, enum, `std::string`) — class-typed fields never produced a default anyway.
- **Node: the `Wrap` constructor compiles for non-assignable classes** — its parameterized-constructor path assigns the freshly built object into the wrapper's inner storage, a statement that failed to compile for a class with deleted copy *and* move assignment even though the emitter registered no such constructor. The path is now `if constexpr`-guarded on assignability (matching the emitter's gate, `GenClass::copy_or_move_assignable`), so such a class keeps its default constructor and everything else compiles out.
- **Node: wrapped class arguments are handed to C++ by reference, not by copy** — `from_napi<T>` now returns `T&` for a wrapped class type (the C++ object living inside the JS object's `Wrap`) instead of a by-value copy. A bound function taking `T&` or a pointer out-parameter therefore mutates the caller-visible JS object, and a type whose copy is shallow or deleted (e.g. a pImpl facade — the copy duplicated the impl pointer and dangled it when the temporary died) no longer breaks. Scalars, strings, enums and vectors still convert by value.
- **WebAssembly (expanded): several build breakers for a real library** — the expanded embind backend now (a) sets `CMAKE_CXX_STANDARD 20` (arch3-style headers use concepts); (b) marshals a by-value class only when it is a *bound* `class_` — an unregistered/incomplete object (`std::array`, `std::initializer_list`, a forward-declared helper returned by value) is skipped instead of aborting the build; (c) emits no constructor for an abstract class (`GenClass::is_abstract`, so embind never tries to allocate one); (d) disambiguates the member-function pointer with an explicit `static_cast` to the exact signature, so an overloaded method no longer trips embind's "no matching function"; (e) emits at most one constructor per arity (embind dispatches constructors by parameter count and throws at registration on a collision — e.g. `Vector3(double)` vs `Vector3(const std::vector&)`). With these, a real library (arch3) builds, links and runs under embind.
- **Node: by-value class marshalling requires a round-trippable type** — the N-API conversion default-constructs a wrapper and copy-assigns, so a class returned/taken by value is now supported only when it is complete, default-constructible and copy-assignable (a non-copyable/incomplete helper like `Model::regions()`'s `ModelRegions` is skipped rather than breaking the build). Pass such types by pointer instead.
- **Python / Node: trampoline override signatures resolve unqualified `std` names** — a trampoline reproduces each virtual's *exact* parameter spelling via `display_string_of`, which drops the `std::` qualifier (e.g. `vector<double, allocator<double>>`), so the generated override failed to compile. The trampoline block now emits a `using namespace std;` scoped to its own `rosetta_py` / `rosetta_node` namespace — enough for the spellings to resolve, with no leak into the binding-registration code or user headers.
- **Python / Node: trampolines skip virtuals they can't marshal** — a virtual method whose signature has no caster (a raw C array, or a pointer the host language can't round-trip — Node-API has no conversion for a bare pointer at all) would make `PYBIND11_OVERRIDE` / `napi_call_override` fail to compile. A new reflection-time `GenMethod::sig_bindable` flag (rejects arrays, all pointers, and `std::vector`-of-pointer — the conservative common denominator across backends) lets the trampoline omit those overrides; a concrete bound class still supplies the body, so the trampoline stays instantiable.
- **Python / nanobind / WebAssembly / Qt / QML: unbindable members no longer break the module** — a method, static method, or field whose return type, a parameter type, or (for a field) its type can't be represented by the target framework now gets skipped instead of aborting the whole build. The two compile-breakers are **raw C arrays** (e.g. a method returning `double (&)[3][3]` — pybind11 / nanobind / embind have no caster, and the Qt / QML backends can't declare `R r` nor round-trip it through `QVariant`) and **types only forward-declared in the binding TU** (pybind needs `sizeof` / `typeid` / `is_polymorphic` on the complete type). A consteval `*_bindable_type` / `*_bindable_fn` guard drops exactly those, matching the support guards the Node, Julia, Java, C#, and REST backends already had. The guard now also covers **constructors** (`py::init<>` / `nb::init<>` / embind `constructor<>` over an unrepresentable parameter no longer aborts the build).
- **Python / nanobind / WebAssembly / Qt / QML: a pointer is judged by its pointee, and `std::vector` by its element** — the previous guard treated any pointer as bindable, but `sizeof(T*)` is always valid so a pointer to an *incomplete* type slipped through and then broke the build inside the framework's caster (it still needs the pointee's `typeid` / `sizeof` to register the conversion). The `*_bindable_type` concept now checks the **pointee** for completeness, and recurses into a `std::vector`'s **element** type (so a `std::vector<T*>` of an incomplete `T` is rejected just like a bare `T*`). A method / field / constructor that names such a type is skipped instead of failing to compile.
- **Python / nanobind / WebAssembly / Julia: the synthesized default constructor was registered behind a runtime `if`** — `bind_*` fell back to registering `T()` with `if (!saw_default_ctor && std::is_default_constructible_v<T>)`. A plain `if` still *instantiates* the discarded branch, so the framework's default-construct (`py::init<>()`, `nb::init<>()`, embind / jlcxx `constructor<>()`, and ultimately `new Trampoline{}`) was compiled even for a non-default-constructible type — a hard error for classes / trampolines whose only constructors take arguments (e.g. a `Surface(Model*, …)` or a pybind trampoline over such a base). Switched to `if constexpr` so the fallback is only instantiated when the type is actually default-constructible. (The Java and C# backends already used `if constexpr`; the Node backend has no such fallback.)
- **Node: reference parameters of bound types** — the N-API backend materialized every argument by value (`from_napi<remove_cvref_t<P>>`), so a non-const lvalue-reference parameter couldn't bind and any function taking a bound type by reference (e.g. an algorithm taking `SurfaceMesh&` and mutating it in place) failed to compile. A new `arg_from_napi<P>` binds reference parameters of wrapped user types directly to the wrapper's persistent `inner` object, so in-place mutations propagate back to the JS object (and `const T&` no longer copies). Applied to free functions, instance/static methods, and constructors.
- **Node: unbindable free functions are skipped, not fatal** — a free function whose signature can't cross the N-API boundary (a pointer out-parameter such as `std::vector<T>*`, a `std::function`, an unsupported return type, …) previously broke the whole module build. `bind_napi_function` now guards on a consteval `napi_free_supported<F>()` and binds a stub that throws on call instead, so the module still loads and every other function stays usable.
- **Template-specialization type names** — `class_name<T>()` no longer hard-errors when a bound free function takes or returns a template specialization (e.g. `pmp::Matrix<float, 3, 1>`, `Eigen::SparseMatrix<double>`); it falls back to the full display spelling when the type has no plain identifier.
- **Operator / conversion members** — the member walk skips functions without an identifier (`operator==`, `operator[]`, `operator T`), which can't be bound by name to a target language and previously broke generation for such classes.
